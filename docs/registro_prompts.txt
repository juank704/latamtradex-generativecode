#RESPUESTA IA:

Por qué ocurre
El SDK de Stripe tipa la propiedad apiVersion como un literal string exacto ligado a la versión del paquete instalado. Al instalar stripe@^17.4.0 (rango ^), npm resolvió una versión menor que ya espera la API '2025-02-24.acacia'. El compilador rechaza el literal antiguo porque no coincide con ese tipo.

Qué cambió y por qué esta solución
Antes: new Stripe(key, { apiVersion: '2024-09-30.acacia' }) → literal no asignable al tipo del SDK actual.
Después: new Stripe(key) → el SDK usa internamente la apiVersion por defecto que coincide con su tipo. Resultado: ningún literal que mantener sincronizado.
Alternativa (si necesitas fijar la versión por gobernanza)
Si tu equipo exige fijar explícitamente la versión de la API por estabilidad, hay dos opciones igualmente válidas:

Fijar la versión del paquete en package.json (sin caret) y dejar el literal en código:


"stripe": "17.4.0"
Usar el literal con as const apuntando al valor que espera el SDK instalado:


new Stripe(key, { apiVersion: '2025-02-24.acacia' as const });
Para tu caso —proyecto académico y modo prueba— la versión sin literal es la más robusta. Vuelve a correr docker compose up --build y el paso de tipos pasará.

Qué pasó
PowerShell se traga las cadenas vacías al pasarlas a procesos externos. Tu --entrypoint "" desapareció, así que Docker leyó app como entrypoint y node como nombre del servicio → "no such service: node".

Solución (PowerShell-safe)
En vez de pasar entrypoint vacío, fuerza sh -c como entrypoint y mete todo el comando como una sola cadena:


docker compose run --rm -p 5555:5555 --entrypoint sh app -c "node /app/node_modules/prisma/build/index.js studio --port 5555 --hostname 0.0.0.0 --browser none --schema /app/prisma/schema.prisma"
Luego abre http://localhost:5555.

Por qué funciona
--entrypoint sh — entrypoint no vacío, PowerShell lo respeta.
app — el servicio.
-c "..." — el argumento que recibe sh (la cadena del comando completo).
Alternativa con --% (stop-parsing de PowerShell)
Si prefieres mantener el comando original tal cual, usa el token --% para que PowerShell deje de tocar los argumentos:


docker compose run --% --rm -p 5555:5555 --entrypoint "" app node /app/node_modules/prisma/build/index.js studio --port 5555 --hostname 0.0.0.0 --browser none --schema /app/prisma/schema.prisma
Cualquiera de las dos formas debería levantar Prisma Studio en localhost:5555. Para cerrarlo, Ctrl + C en la terminal.

⚠️ Sobre migrate dev — por qué usé db push
Ejecutar npx prisma migrate dev aquí habría sido destructivo. El proyecto nunca ha usado migraciones versionadas (no existe carpeta prisma/migrations ni tabla _prisma_migrations); las tablas se crearon con db push. Al correr migrate dev contra esa base, Prisma detecta drift, asume que el estado no le pertenece y propone resetear la base (borrando tus 8 productos, usuarios y la cotización). Además, en esta imagen standalone los archivos de migración no persistirían (filesystem efímero, /app/prisma no es volumen). Por eso apliqué el cambio con db push (el mecanismo real del docker-entrypoint.sh), que añade las columnas con sus defaults de forma no destructiva. Si quieres migraciones versionadas para el repo, hay que baselinear primero — te lo monto en un paso aparte si lo necesitas.
